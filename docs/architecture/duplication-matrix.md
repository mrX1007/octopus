# OCTOPUS duplication and migration matrix

Status: current-system inventory with Waves 4–6 resolution ledger

Baseline: working tree (2026-07-15)
Scope: first-party Python code; `tests/`, `venv/`, generated artifacts and both dirty vendor submodules are excluded.

This document records existing ownership and safe extraction seams. It is not a rewrite proposal. In particular, it preserves `FactStore` as the evidence source of truth, `PipelineRuntime` as the intended runtime I/O boundary, and `ExecutionPolicy` as the final execution authority.

## How to read the matrix

- **Current owner** is the component that is authoritative today, even when callers bypass it.
- **Compatibility constraint** is observable behavior that cannot disappear during extraction.
- **Migration seam** is the smallest change that can remove duplication without creating a second subsystem.
- **Delete precondition** must be satisfied before the duplicate or facade can be removed.
- “No in-repo caller” does not prove that a public facade is unused by external plugins or scripts.

## G. Broad exception swallowing

The inventory command found **349** first-party `except Exception`, `except BaseException`, or `suppress(Exception)` sites. The count is a navigation aid, not 349 defects: worker/process boundaries legitimately serialize failures. The critical duplication is that every subsystem invents its own “error becomes empty/string/False” contract.

| Boundary | Current owner and callers | Compatibility constraint | Migration seam | Delete precondition |

## Remaining no-parallel-system migration order

Canonical execution results, runtime adapters, entity identity/graph projection,
action lifecycle, provider fallback, recovery, evidence reports and decision
observability are complete through their existing owners. Remaining work should
proceed in this order:

1. Move the last orchestration-specific parse/persist hooks through the existing
   runtime ingress one side effect at a time.
2. Consolidate URL/target/status helpers behind the established canonical
   identity/execution functions while preserving facades.
3. Generate CLI menu/availability from the concrete registry without merging
   planner vocabulary into callable metadata.
4. Route remaining legacy/manual workflows and application/C2 process creation
   through their appropriate action or lifecycle boundary.
5. Remove import-time migration/bootstrap and mutable service globals through
   one composition root.
6. Delete public aliases/wrappers only after contract tests, repository/plugin
   usage audit, telemetry/deprecation window and migration documentation satisfy
   each row's precondition.
